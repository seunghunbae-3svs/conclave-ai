# v0.13.8 — GNU `patch -p1 --fuzz=3` fallback

Released **2026-04-27**.

## TL;DR

The first live closing-cycle attempt on `eventbadge#29` (sha `279cb22`) surfaced a real RC: the worker emitted a unified diff with the hunk header line number off by one — `@@ -17,...` against a deletion target at line 18. `git apply --check --recount` rejected on the Linux runner.

`--recount` only recomputes line *counts*, not start positions. On Linux runners (same git 2.53.0 as Windows), the offset tolerance wasn't enough.

Fix: both apply paths in autofix now fall back to GNU `patch -p1 --fuzz=3 -F 3`:
1. `runPerBlocker` (per-blocker validate) — uses `--dry-run`.
2. `autofix.ts` post-loop apply — actual write.

Both must succeed for a fix to land. If `patch(1)` is unavailable (Windows runners) or also rejects, the existing conflict-with-diagnostic path runs.

2 new tests cover the fallback-success-path and both-fail-path.
